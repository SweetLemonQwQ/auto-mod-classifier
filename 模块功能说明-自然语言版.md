# 自动筛选模组分类器模块功能说明

这份说明不是写给程序员看的，而是写给“未来的你”看的。

你可以把现在这个项目理解成四层：

1. `ui`
这层就是界面层。
它负责按钮、输入框、弹窗、日志框这些你看得见的东西。
它不负责真正判断模组，也不负责真正制作服务端。

2. `application`
这层就是“任务入口层”。
它的作用是把“用户想做什么”说清楚，比如：
“我要筛模组”
“我要一键开服”
然后把任务交给下面真正干活的模块。

3. `infrastructure`
这层就是“接地气的适配层”。
它负责把现实里的输入接进来，比如目录、以后要加的 `mrpack`、整合包 `zip`。
也负责把现在还能用的旧逻辑包装起来，先让整个项目稳定运行。

4. `classifier` 和 `server_builder`
这两层就是核心业务。
`classifier` 负责模组筛选。
`server_builder` 负责一键制作服务端。


## 根目录下这些文件是干什么的

### `自动筛选模组分类器.pyw`
这是程序入口。
你双击启动时，真正先跑的是它。
它本身不做业务，只是把程序带到 `ui` 里去。

### `auto_mod_classifier/__init__.py`
这是主包的标记文件。
可以理解成告诉 Python：“这一整个文件夹是一个完整模块。”

### `auto_mod_classifier/shared.py`
这是全项目通用的公共定义。
里面放的是很多地方都会用到的东西，比如：

- 模组信息长什么样
- 分类结果长什么样
- 版本候选长什么样
- Java 运行时信息长什么样
- 一些通用常量和工具函数

简单说，它像整个项目的“公共词典”。

### `auto_mod_classifier/bootstrap.py`
这是“装配中心”。
它决定程序默认用哪套组件拼起来运行。

比如它会决定：

- 当前输入源默认支持哪些格式
- 模组筛选默认用哪套服务
- 一键开服默认用哪套服务

如果以后你想换前端、换输入源、换筛选服务，这里就是最重要的拼装点之一。

### `auto_mod_classifier/tasks.py`
这是界面层和业务层之间的转接层。
它的工作很像翻译：

- 把界面传进来的参数
- 翻译成应用层能理解的任务请求

它自己不做判断逻辑，也不做开服逻辑。
它只是负责把任务送对地方。


## `application` 目录

这一层的重点不是“怎么做”，而是“我要做什么”。

### `application/models.py`
这里放任务请求模型。
比如：

- 筛模组任务需要哪些参数
- 一键开服任务需要哪些参数
- 导入器整理完输入后，要交给后面什么数据

你可以把它理解成“任务表单”和“标准输入格式”。

### `application/contracts.py`
这里放接口约定。
说白了就是先把规矩定好：

- 导入器最少要会干什么
- 模组筛选服务最少要会干什么
- 一键开服服务最少要会干什么

这样以后你换实现的时候，不至于全项目一起乱。

### `application/importers.py`
这是输入源选择器。
它负责判断用户给的东西，到底该交给哪个导入器处理。

比如以后支持：

- 普通目录
- `mrpack`
- 整合包 `zip`

就是从这里挑选对应处理器。

### `application/use_cases.py`
这是应用层最核心的两个任务入口：

- `ScanModsUseCase`
- `BuildServerUseCase`

你可以把它理解成两个总按钮：

- 一个按钮叫“开始筛模组”
- 一个按钮叫“开始一键开服”

前端以后无论是 Tk、网页、还是命令行，理论上都应该优先调这里，而不是直接去碰下面复杂的业务细节。


## `infrastructure` 目录

这一层可以理解成“现实世界适配层”。
它负责把各种外部输入和现有旧逻辑接进来。

### `infrastructure/importers.py`
这里放各种输入源导入器。

现在已经有：

- `DirectorySourceImporter`

也预留了：

- `MrpackSourceImporter`
- `ZipModpackSourceImporter`

意思就是后面你真要加 `mrpack` 或 `zip`，不用大动主结构，直接在这里补实现就行。

### `infrastructure/legacy_services.py`
这里是“旧逻辑适配器”。

因为你原来那套逻辑是能用的，所以这层做的事不是推倒重写，而是把旧逻辑包起来，让它能接进现在这套新架构。

简单说：

- 外面看起来是新结构
- 里面暂时还能复用旧能力

这样改造风险小很多。


## `classifier` 目录

这是模组筛选核心。
现在已经不是以前那种“全塞在一个大类里”的状态了。

### `classifier/core.py`
这是筛选运行时核心。
它现在主要保留的是和运行时有关的能力，比如：

- 网络请求节流
- 浏览器相关能力
- 文本处理基础能力

它不再承担“所有判断逻辑都写在这里”的角色。

### `classifier/contracts.py`
这是筛选系统内部的规则约定。
它规定了几类角色：

- 元数据读取器
- 本地分类器
- 远程分类来源
- 分类策略

你以后想换筛选方式，先看这里，理解每个角色是干什么的。

### `classifier/models.py`
这里放筛选阶段自己的小模型。

比如：

- 当前这次筛选开了哪些开关
- 某个远程来源返回了什么结果

它相当于是筛选核心内部的小型数据结构。

### `classifier/text_utils.py`
这是文字处理工具箱。
它负责很多“像但不完全一样”的字符串处理工作，比如：

- 文件名清洗
- 标题比对
- 模组名相似度判断
- 查询词扩展

模组筛选为什么难，很大一部分难点其实都在这里。

### `classifier/browser.py`
这是浏览器相关能力。
它主要是为了处理像 MC百科、CurseForge 这种不老实的网页环境。
比如验证码、Cloudflare、真实浏览器访问这些事，都在这块。

### `classifier/network.py`
这是网络访问能力。
它负责：

- 普通请求
- 缓存
- 限流
- 并发请求协调

可以理解成筛选系统的“外网通信层”。

### `classifier/services.py`
这是现在最重要的筛选业务模块之一。
里面已经拆成了几种清晰角色：

- `JarMetadataReader`
  负责从 jar 里读信息

- `DefaultLocalClassifier`
  负责本地规则判断

- `ModrinthRemoteSource`
  负责查 Modrinth

- `McmodRemoteSource`
  负责查 MC百科

- `CurseforgeRemoteSource`
  负责查 CurseForge

- `DefaultClassificationStrategy`
  负责决定“先查谁、后查谁、查不到怎么办”

你以后要改筛选策略，最值得先看的就是这个文件。

### `classifier/pipeline.py`
这是筛选流程编排器。
它负责把整个筛选真正跑起来。

你可以把它理解成流水线调度员：

- 先让每个 jar 读元数据
- 先走本地判断
- 本地判断不够再去远程补查
- 控制并发
- 汇总最终结果

它不关心某个站点怎么查，它关心的是“整个筛选流程怎么跑”。

### `classifier/__init__.py`
这是筛选模块的统一出口。
外面如果要用筛选核心，尽量从这里拿，而不是到处乱 import 里面的细节文件。


## `server_builder` 目录

这是“一键制作服务端”的核心。
现在这块也已经从原来那种“大类继承一串 mixin”的结构，改成了更好维护的组合式结构。

### `server_builder/context.py`
这是开服流程的共享运行时状态。
比如：

- 当前用哪个分类器
- 当前日志怎么写
- 当前界面怎么回传状态
- 用户选择了哪些开关

它像开服流程所有服务公用的“背包”。

### `server_builder/common.py`
这是开服流程的公共工具层。
比如：

- 写日志
- 阶段进度更新
- 下载网络内容
- 一些通用路径判断

它的意义是把所有服务都要重复用的小工具收口起来。

### `server_builder/core.py`
这是一键开服总入口。
它本身现在不再自己干全部事情，而是负责把下面这些服务组装起来。

你可以把它理解成“总控制台”，不是“全能打工人”。

### `server_builder/services.py`
这是开服核心服务集合。
里面是现在开服链的主体。

主要分成几块：

- `ServerVersionService`
  负责识别客户端版本、加载器类型、安装器信息

- `ServerJavaService`
  负责找 Java、验 Java、判断该用哪个版本

- `ServerInstallService`
  负责执行服务端安装器，以及开服过程里对 mods 做自动筛选

- `ServerModService`
  负责模组复核、配置目录复核、复制文件

- `ServerLaunchService`
  负责生成启动脚本、首次启动、修配置、再次启动验证

- `ServerReportingService`
  负责输出报告和日志

- `ServerWorkflowService`
  负责把整个一键开服步骤串起来

如果你以后想改“一键开服流程顺序”，优先看 `ServerWorkflowService`。
如果你想改某个具体阶段，就看对应那个服务。

### `server_builder/__init__.py`
这是一键开服模块的统一出口。


## `ui` 目录

这一层就是当前桌面版界面。

### `ui/app.py`
这是主窗口。
里面有：

- 两个标签页
- 路径选择
- 任务启动
- 日志显示
- 进度更新

它的工作重点是“和用户交互”，不是做业务判断。

### `ui/dialogs.py`
这是各种弹窗。
比如：

- 版本选择弹窗
- 模组复核勾选弹窗
- 配置目录复核弹窗

### `ui/common.py`
这是界面层自己的小公共模块。
目前主要放界面面板状态这类只和 Tk 有关的结构。

### `ui/__init__.py`
界面模块统一出口。


## 如果你以后要改功能，优先看哪里

### 如果你想支持 `mrpack`
先看：

- `infrastructure/importers.py`
- `application/importers.py`

### 如果你想支持整合包 `zip`
先看：

- `infrastructure/importers.py`
- `application/models.py`

### 如果你想改模组筛选策略
先看：

- `classifier/services.py`
- `classifier/pipeline.py`
- `classifier/contracts.py`

### 如果你想加一个新的远程来源
先看：

- `classifier/services.py`
- `classifier/contracts.py`

### 如果你想改一键开服流程顺序
先看：

- `server_builder/services.py` 里的 `ServerWorkflowService`

### 如果你想改单独某一步，比如 Java 检测
先看：

- `server_builder/services.py` 里的 `ServerJavaService`

### 如果你想换前端，比如以后做网页
先看：

- `application/use_cases.py`
- `tasks.py`
- `bootstrap.py`

因为现在前端已经和核心逻辑分开了，网页前端不需要重写整套业务。


## 你可以怎么理解现在这套结构

以前更像是：

- 一个巨大的脚本
- 或者几个大类互相缠在一起

现在更像是：

- `ui` 负责跟人说话
- `application` 负责接任务
- `infrastructure` 负责接现实输入
- `classifier` 负责筛模组
- `server_builder` 负责做服务端

这样以后你不管是继续做功能，还是单独重写某一块，都会轻松很多。
